# 故事板设计逻辑

本文件定义 skill 的导演规划层。

故事板设计不是把用户一句话扩写成长 prompt，而是把需求拆成一套可执行的视觉决策：

- 拍什么
- 先拍什么后拍什么
- 从哪里拍
- 每一格承担什么功能
- 什么必须一致
- 什么绝对不能串位

## 产物要求

在进入渲染器前，必须先形成 `storyboard_plan`。

推荐结构：

```json
{
  "board_type": "",
  "output_format": "",
  "panel_count": 0,
  "camera_strategy": "",
  "continuity_rules": [],
  "panels": [],
  "risk_controls": []
}
```

## 第一步：识别用户真正想解决的视觉问题

先判断用户想要的到底是哪一种视觉控制：

| 目标类型 | 说明 |
|---|---|
| 动作过程 | 强调身体动作的开始、中段、高点、结束 |
| 空间建立 | 强调外到内、远到近、空间关系 |
| 产品交互 | 强调产品或道具如何被看见、拿起、使用 |
| 转场变化 | 强调状态 A 到状态 B 之间的可视化桥梁 |
| 情绪表演 | 强调脸部、眼神、姿态和情绪递进 |
| 机位探索 | 强调同一主体在不同距离和角度下的呈现 |

如果目标在两种完全不同的类型之间摇摆，必须补问，不能硬猜。

## 第零步：补全故事字段（素材锁定后）

在进入分镜类型选择前，必须先形成完整的叙事控制字段：

| 字段 | 来源优先级 |
|------|-----------|
| `story_framework` | 用户必填 |
| `main_action` | 用户兼容输入 > 等于 `story_framework` |
| `scene_description` | 用户输入 > 场景参考图 > 导演推断 |
| `performance_focus` | 用户输入 > 角色/产品等参考图 + 故事 > 导演推断 |

Agent 执行时必须读取 `scripts/infer_story_fields.py` 的逻辑，或在脑中等价执行，并在 `storyboard_request.inferred_fields` 中标注推断来源。

## 第二步：建立素材锁定图

根据 `references/03_asset_classification.md`，把所有素材映射为清晰职责。

最少要明确三件事：

1. **谁定义身份**
2. **谁定义空间**
3. **谁只提供风格、服装、光照、运动或布局**

同时为每个素材写出：

- 必须继承什么
- 禁止继承什么

## 第三步：选择分镜类型

根据目标和素材，选择最适合的 `board_type`。

规则：

- 连续动作、单一机位推进：优先 `continuous_shot_board`
- 有明确动作阶段：优先 `action_progression_board`
- 重点是建立地点与路径：优先 `spatial_establishing_board`
- 重点是展示产品或道具：优先 `product_interaction_board`
- 重点是状态变化：优先 `transition_board`
- 重点是情绪微表演：优先 `emotion_performance_board`
- 同一场景多角度切换：优先 `multi_shot_board`

不允许无理由默认某一种类型。

## 第四步：确定 panel 数量

panel 数量由信息复杂度决定，不由模板决定。

简化规则：

- 静态或单段动作：1 到 2 格
- 有清晰起止：2 到 3 格
- 有递进节拍：3 到 5 格
- 有复杂动作或机位变化：5 到 8 格

禁止两种错误：

- 为了好看硬凑格数
- 为了省事压缩到看不清动作

## 第五步：给每格分配导演功能

每一格都必须有存在理由。

每格至少回答：

- 这一格负责传达什么新信息
- 与前一格相比推进了什么
- 这一格最关键的镜头控制点是什么

常用功能：

- 建立地点
- 建立空间关系
- 建立主体位置
- 展示动作起点
- 展示动作中段
- 展示动作高点
- 展示动作收尾
- 展示表情变化
- 展示产品交互
- 展示转场中间态
- 展示机位变化
- 展示尾帧停留

## 第六步：形成连续性规则

在真正写提示词前，必须先写连续性规则。

至少覆盖：

- 主体身份连续
- 场景结构连续
- 道具或产品连续
- 光向与时间连续
- 屏幕方向与运动方向连续

如果提供了上一帧素材，则默认连续性优先级最高。

## 第七步：再进入提示词生成

只有当以下对象都稳定后，才进入提示词生成：

- `storyboard_request`
- `asset_lock_map`
- `storyboard_plan`

否则很容易出现下面三种假智能：

- 只会复述用户原话
- 会写漂亮话但不会导戏
- 会生图但素材一致性很差
